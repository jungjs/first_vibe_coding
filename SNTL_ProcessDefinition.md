## P03. 로그인 / 토큰 갱신 / 로그아웃

> **도메인:** 회원/인증 | **처리주체:** 회원
> JWT 기반 인증 흐름. 로그인 → Access/Refresh Token 발급 → 갱신 → 로그아웃.

| 단계 | 처리주체 | 활동명 | 상세 처리 내용 | 입력 | 출력/결과 | 판단/분기 | 예외처리 | 비고 |
|---|---|---|---|---|---|---|---|---|
| 1 | 회원 | 로그인 요청 | 로그인ID·비밀번호 입력 후 로그인 버튼 클릭 | loginId, password | 로그인 요청 | - | - | - |
| 2 | 시스템 | 계정 존재 확인 | DB에서 loginId 조회 | loginId | 계정 존재 여부 | 존재 → 3단계 / 미존재 → 오류 | '아이디 또는 비밀번호가 틀렸습니다' 안내 (계정 존재 여부 미노출) | 보안상 미구분 |
| 3 | 시스템 | 비밀번호 검증 | BCrypt로 입력 PW와 저장된 해시 비교 | 입력 PW, 해시 PW | 검증 결과 | 일치 → 4단계 / 불일치 → 오류 | **5회 실패 시 계정 잠금(30분)** | - |
| 4 | 시스템 | 계정 상태 확인 | 회원 status 확인 (ACTIVE 여부) | status 값 | 상태 확인 결과 | ACTIVE → 5단계 / 비활성 → 로그인 거부 | 법인 미승인·탈퇴 계정 안내 | - |
| 5 | 시스템 | JWT 토큰 발급 | Access Token(30분)·Refresh Token(7일) 생성. Refresh Token은 Redis에 저장 | 회원 정보 | Access Token, Refresh Token | - | Redis 저장 실패 시 로그인 실패 처리 | - |
| 6 | 회원 | API 호출 | Access Token을 Authorization 헤더에 담아 API 요청 | Access Token | API 응답 | 유효 → 정상처리 / 만료 → 7단계 | 토큰 위변조 → 401 반환 | - |
| 7 | 시스템 | 토큰 갱신 | Access Token 만료 시 Refresh Token으로 재발급 요청 | Refresh Token | 신규 Access Token | Redis에 존재·유효 → 재발급 / 없음·만료 → 재로그인 요청 | Refresh Token 블랙리스트 확인 | - |
| 8 | 회원 | 로그아웃 요청 | 로그아웃 버튼 클릭 | - | 로그아웃 요청 | - | - | - |
| 9 | 시스템 | 토큰 무효화 | Redis에서 Refresh Token 삭제. Access Token을 블랙리스트(Redis)에 등록(잔여 유효시간 동안) | Access Token, Refresh Token | 토큰 무효화 완료 | - | Redis 실패 시 로그 기록 | - |

**핵심 보안 규칙:**
- 비밀번호 5회 실패 → 계정 잠금 30분
- 로그아웃 시 Access Token 블랙리스트 등록 (잔여 유효시간 유지)
- 계정 존재 여부는 로그인 실패 메시지에서 미노출 (보안)

## P04. 회원탈퇴 (Soft Delete)

> **도메인:** 회원/인증 | **처리주체:** 회원
> 개인/법인회원이 직접 탈퇴 신청. 개인정보는 Null 처리, ID는 보존하는 Soft Delete 방식.

| 단계 | 처리주체 | 활동명 | 상세 처리 내용 | 입력 | 출력/결과 | 판단/분기 | 예외처리 | 비고 |
|---|---|---|---|---|---|---|---|---|
| 1 | 회원 | 탈퇴 신청 | 마이페이지에서 '회원탈퇴' 메뉴 선택. 탈퇴 사유 입력(선택) | 탈퇴 사유 | 탈퇴 확인 화면 | - | - | - |
| 2 | 회원 | 본인 재확인 | 비밀번호 재입력으로 본인 확인 | 비밀번호 | 검증 결과 | 일치 → 3단계 / 불일치 → 오류 | 3회 실패 시 탈퇴 절차 종료 | - |
| 3 | 시스템 | 처리 가능 여부 확인 | 미결제 청구서·진행 중 오더 존재 여부 확인 | 회원ID | 확인 결과 | 이상 없음 → 4단계 / 미결제·진행중 → 탈퇴 보류 안내 | '미결제 청구서 정산 후 탈퇴 가능' 안내 | - |
| 4a | 시스템 | 개인회원 Null 처리 | 이름·생년월일·휴대폰·주소·이메일 → NULL / status = WITHDRAWN / 로그인ID만 보존 | 회원 레코드 | Null 처리 완료 | - | DB 오류 시 롤백 | 개인회원 경로 |
| 4b | 시스템 | 법인회원 Null 처리 | 법인: 법인명 제외 개인정보 NULL / 법인관리자·부서·부서관리자 개인정보 NULL / 법인증빙파일 MinIO에서 완전 삭제 | 법인 관련 레코드 전체 | Null 처리 및 파일 삭제 완료 | - | 파일 삭제 실패 시 별도 로그 기록 | 법인회원 경로 |
| 5 | 시스템 | JWT 토큰 강제 만료 | Redis의 Refresh Token 즉시 삭제 및 Access Token 블랙리스트 등록 | - | 토큰 무효화 | - | - | - |
| 6 | 시스템 | 탈퇴 완료 처리 | 탈퇴 완료 안내 이메일 발송(가능한 경우). 로그아웃 처리 후 메인 화면으로 이동 | - | 완료 안내 | - | 이메일 NULL인 경우 발송 생략 | - |

**Soft Delete 규칙:**
- 개인회원: 이름·생년월일·휴대폰·주소·이메일 → NULL, login_id 보존
- 법인회원: 법인 관련 전체 개인정보 NULL, 법인증빙파일 MinIO 완전 삭제

| 9 | 시스템 | 잔액 차감 (승인 시) | 회원 balance -= refund_amount. 잔액 0 이하 방지 검증 재수행 | 환불 금액 | 잔액 갱신 완료 | - | 잔액 부족 시 롤백 및 상태 원복 | 트랜잭션 처리 |

**핵심 규칙:**
- 환불 요청 등록 시 잔액을 즉시 차감하지 않음 (관리자 CONFIRMED 후 차감)

- 잔액 차감 시 트랜잭션 처리 필수

## P16. 환율 조회 및 서울외국환중개소 연계

> **도메인:** 기초정보관리 | **처리주체:** 시스템·관리자
> 서울외국환중개소 API를 통해 영업일 기준 환율을 자동 연계하고 DB에 저장하는 절차.
> 배치 자동 연계(영업일 오전 8시)와 관리자 수동 연계 두 가지 경로.

| 단계 | 처리주체 | 활동명 | 상세 처리 내용 | 입력 | 출력/결과 | 판단/분기 | 예외처리 | 비고 |
|---|---|---|---|---|---|---|---|---|
| 1a | 시스템(배치) | 배치 스케줄러 실행 | 매 영업일 오전 8시 배치 스케줄러 실행. 자동 환율 연계 시작 | 스케줄 트리거 | 배치 실행 | 영업일 → 2단계 / 비영업일(공휴일·주말) → 실행 생략 | 공휴일 판단: 내장 달력 또는 공공 API 활용 | 자동 경로 |
| 1b | 관리자 | 수동 연계 실행 | 관리자 화면 > '기초정보관리' > '환율 관리' > '환율 연계' 버튼 클릭 | 수동 실행 요청 | 연계 실행 | - | - | 수동 경로 |
| 2 | 시스템 | 당일 환율 기등록 여부 확인 | exchange_rate 테이블에서 rate_date = 오늘(영업일) 데이터 존재 여부 확인 | rate_date | 기등록 여부 | 기등록 없음 → 3단계 / 이미 존재 → 갱신 여부 판단 | 수동 연계 시: 기존 데이터 덮어쓰기 확인 팝업 표시 | - |

| 4 | 시스템 | 응답 데이터 파싱 | API 응답 JSON 파싱. 통화코드·통화명·기준환율 추출 | API 응답 JSON | 파싱된 환율 목록 | 파싱 성공 → 5단계 / 파싱 실패 → 오류 로그 및 관리자 알림 | API 스펙 변경 시 파싱 오류 발생 가능 | - |

| 6 | 시스템 | 연계 완료 처리 | 연계 성공 일시 기록. 배치 실행 이력 저장 | - | 완료 처리 | - | - | - |

**핵심 규칙:**
- 연계 기준: **영업일 오전 8시** (서울외국환중개소 고시 기준)
- API 타임아웃: **10초**, 재시도: **3회**
- 저장 방식: 통화코드 + 기준일 기준 UPSERT (중복 저장 방지)
- 관리자 수동 수정 시 수정 이력 별도 기록
